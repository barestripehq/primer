# Contributing to primer

## Development setup

**Prerequisites:** Rust toolchain via [rustup](https://rustup.rs).

```sh
git clone https://github.com/barestripehq/primer
cd primer
cargo build                      # default build, no AI
cargo build --features ai        # build with AI summary layer
```

## Running tests

```sh
cargo test                       # full test suite
cargo test --features ai         # include AI-gated tests
```

## Linting

```sh
cargo fmt --check                # check formatting
cargo clippy -- -D warnings      # lint (warnings as errors)
```

Fix formatting in place with `cargo fmt`.

## Project structure

```
src/
  main.rs          — CLI entry point (clap)
  shim.rs          — argv parsing, shim dispatch, pass-through logic
  manifest.rs      — manifest file parsers (diff and full-file)
  config.rs        — ~/.primer/config.toml read/write
  cache.rs         — 24h local cache keyed by ecosystem+package+version
  engine/
    osv.rs         — OSV API client
  cli/
    hook.rs        — `primer hook install` / `primer hook check`
    init.rs        — shim setup and teardown
    scan.rs        — interactive scan flow
  summary/         — AI summary layer (feature-gated)
    download.rs    — model download via hf-hub
    infer.rs       — GGUF inference via candle
```

## Releasing

primer uses [`cargo-release`](https://github.com/crate-ci/cargo-release) to cut releases. Binaries are built and distributed by [cargo-dist](https://opensource.axo.dev/cargo-dist/) via GitHub Actions.

### One-time setup

```sh
cargo install cargo-release
```

### Cutting a release

| Change type | Command |
|-------------|---------|
| Bug fix | `cargo release patch` |
| New feature (backwards-compatible) | `cargo release minor` |
| Breaking change | `cargo release major` |

`cargo release` will:
1. Bump the version in `Cargo.toml`
2. Commit with `chore: release <version>`
3. Create an annotated git tag `v<version>`
4. Push the commit and tag to `origin`

The pushed tag triggers the GitHub Actions release workflow, which cross-compiles binaries for all five targets and publishes a GitHub Release with shell and PowerShell installers.

### What not to do

- Do not edit the version in `Cargo.toml` manually before releasing — `cargo release` handles it.
- Do not push a tag manually — let `cargo release` create it so it stays in sync with the `Cargo.toml` bump commit.
- Do not edit `.github/workflows/release.yml` by hand — it is generated by `cargo dist generate` and validated at runtime.

## CI

Two workflows run on every push to `main` and on pull requests:

| Workflow | File | What it checks |
|----------|------|---------------|
| CI | `.github/workflows/ci.yml` | `cargo test`, `cargo fmt`, `cargo clippy` on ubuntu / macOS / Windows |
| Release | `.github/workflows/release.yml` | Triggered by `v*` tags only — builds and publishes binaries |
